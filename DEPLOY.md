## Крок 1 — Що підготувати

Перед деплоєм приготуйте 3 речі:

### 1.1. Пароль для Gmail (App Password) — щоб приходили заявки
1. Зайдіть у Google-акаунт, з якого хочете надсилати листи.
2. Увімкніть двофакторну автентифікацію: <https://myaccount.google.com/security>
3. Створіть пароль додатка: <https://myaccount.google.com/apppasswords>
4. Скопіюйте 16 символів — це буде `SMTP_PASS`.

> Можна використати будь-яку іншу пошту з SMTP (Outlook, Web.de, власний домен) —
> тоді змініть `SMTP_HOST` / `SMTP_PORT` / `SMTP_USER` / `SMTP_PASS`.

### 1.2. Надійний пароль для адмін-панелі
Придумайте пароль для входу в `/admin` (НЕ «admin»). Це буде `ADMIN_PASSWORD`.

### 1.3. Секретний ключ `AUTH_SECRET`
Згенеруйте випадковий рядок. У терміналі (де є Node) виконайте:
```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```
Скопіюйте отриманий рядок — це буде `AUTH_SECRET`.

## Крок 2 — Змінні середовища (.env)

Сайт налаштовується через змінні середовища. **Локально** вони лежать у файлі
`server/.env` (скопіюйте з `server/.env.example`). **На хостингу** ті самі ключі
додаються в розділі «Environment / Variables» сервісу — файл `.env` туди не
завантажується (він у `.gitignore`).

| Змінна | Обовʼязкова | Що вписати |
|--------|:-----------:|------------|
| `NODE_ENV` | так | `production` |
| `PORT` | ні | На хостингу **не задавайте** — платформа підставить свій. Локально `3000`. |
| `ADMIN_EMAIL` | так | Пошта, **куди приходять заявки** (напр. `radarix75@gmail.com`) |
| `SMTP_HOST` | так | `smtp.gmail.com` (для Gmail) |
| `SMTP_PORT` | так | `587` |
| `SMTP_USER` | так | Повна адреса пошти-відправника (напр. `school.mail@gmail.com`) |
| `SMTP_PASS` | так | 16-символьний App Password з кроку 1.1 |
| `MAIL_FROM_NAME` | ні | Імʼя відправника (напр. `IT Kinderschule`) |
| `SEND_CONFIRMATION` | ні | `true` — щоб батькам приходив лист-підтвердження |
| `ADMIN_LOGIN` | так | Логін для `/admin` (напр. `admin`) |
| `ADMIN_PASSWORD` | так | Ваш надійний пароль (крок 1.2) |
| `AUTH_SECRET` | так | Випадковий рядок (крок 1.3) |
| `DATA_DIR` | див. крок 7 | Шлях до постійного диска для даних (напр. `/var/data`) |
| `UPLOAD_DIR` | див. крок 7 | Шлях для завантажених фото (напр. `/var/data/uploads`) |

> Сайт запуститься **навіть без пошти** — заявки все одно збережуться в адмінці,
> просто не надсилатимуться листи. Але `ADMIN_PASSWORD` і `AUTH_SECRET` задайте
> обовʼязково, інакше при `NODE_ENV=production` сервер виведе попередження.

## Крок 3 — Деплой на Render (рекомендовано)

[Render.com](https://render.com) — простий, має безкоштовний тариф.

### 3.1. Залийте код на GitHub
1. Створіть репозиторій на <https://github.com/new>.
2. Завантажте туди вміст папки проєкту (через GitHub Desktop або командами):
   ```bash
   cd "C:/Valko Volodia/java/IT Kinderschule"
   git init
   git add .
   git commit -m "IT Kinderschule"
   git branch -M main
   git remote add origin https://github.com/ВАШ-ЛОГІН/it-kinderschule.git
   git push -u origin main
   ```
   > Файли `.env`, `node_modules`, `data/` не потраплять у git — так і має бути.

### 3.2. Створіть Web Service на Render
1. Render → **New** → **Web Service** → підключіть свій GitHub-репозиторій.
2. Налаштування:
   - **Runtime:** `Node`
   - **Build Command:** `npm install`
   - **Start Command:** `npm start`
   - **Health Check Path:** `/healthz`
3. У розділі **Environment** додайте всі змінні з [кроку 2](#крок-2--змінні-середовища-env)
   (без `PORT` — Render задасть його сам).
4. Натисніть **Create Web Service**. За 2–3 хв сайт буде онлайн за адресою
   `https://назва.onrender.com`.

### 3.3. Підключіть постійний диск (щоб дані не зникали)
Дуже важливо (див. [крок 7](#-збереження-даних--найважливіше)):
1. У сервісі → вкладка **Disks** → **Add Disk**.
2. **Mount Path:** `/var/data`, розмір — напр. 1 GB.
3. Додайте змінні: `DATA_DIR=/var/data` та `UPLOAD_DIR=/var/data/uploads`.
4. Збережіть — сервіс перезапуститься, і всі заявки/контент/фото зберігатимуться на диску.

> На безкоштовному тарифі постійний диск може бути недоступний — тоді розгляньте
> платний план Render або Railway з томом. Без диска дані скидаються при кожному оновленні.

## Альтернатива: Railway

[Railway.app](https://railway.app):
1. **New Project** → **Deploy from GitHub repo** → виберіть репозиторій.
2. Railway сам визначить Node. Start command за потреби: `npm start`.
3. **Variables** → додайте змінні з [кроку 2](#крок-2--змінні-середовища-env).
4. **Volumes** → додайте том, наприклад на `/data`, і змінні
   `DATA_DIR=/data`, `UPLOAD_DIR=/data/uploads`.
5. **Settings** → **Generate Domain**, щоб отримати публічну адресу.

## Альтернатива: власний сервер (VPS)

На сервері з Ubuntu (з установленим Node 18+):
```bash
# 1. Завантажте код
git clone https://github.com/ВАШ-ЛОГІН/it-kinderschule.git
cd it-kinderschule

# 2. Встановіть залежності
npm install

# 3. Створіть і заповніть .env
cp server/.env.example server/.env
nano server/.env        # впишіть свої значення

# 4. Запустіть через pm2 (працюватиме у фоні й після перезавантаження)
npm install -g pm2
pm2 start server/server.js --name it-kinderschule
pm2 save
pm2 startup
```
Далі налаштуйте Nginx як reverse-proxy на порт 3000 і HTTPS через Let's Encrypt (`certbot`).
Дані вже зберігаються локально в теці `data/` — резервно копіюйте її.

## Альтернатива: Docker / Fly.io

У проєкті є готовий `Dockerfile`.
```bash
# Локально:
docker build -t it-kinderschule .
docker run -p 3000:3000 --env-file server/.env -v itk_data:/data it-kinderschule
```
Для [Fly.io](https://fly.io): `fly launch` (визначить Dockerfile) → задайте секрети
`fly secrets set ADMIN_PASSWORD=... AUTH_SECRET=... SMTP_USER=... SMTP_PASS=...` →
створіть том `fly volumes create itk_data --size 1` (монтується у `/data`).

## ⚠️ Збереження даних — найважливіше

Заявки, підписники, контент і завантажені фото зберігаються у **файлах**:
- `data/applications.json`, `data/subscribers.json`, `data/content.json`
- завантажені фото — у `public/images/uploads/`

На більшості хмарних хостингів файлова система **тимчасова**: при кожному
передеплої/перезапуску вона **обнуляється**. Щоб НЕ втратити заявки й фото:

1. Підключіть **постійний диск (volume)** — як описано вище.
2. Вкажіть змінні `DATA_DIR` і `UPLOAD_DIR`, що ведуть на цей диск
   (напр. `DATA_DIR=/var/data`, `UPLOAD_DIR=/var/data/uploads`).

Сервер сам створить потрібні файли на диску при першому запуску.
**VPS** зберігає файли локально й цієї проблеми не має (просто робіть бекап теки `data/`).

## ✅ Після деплою — чекліст

- [ ] Сайт відкривається за публічною адресою (`https://...`).
- [ ] `/healthz` повертає `{"ok":true}`.
- [ ] Зайшли в `/admin` зі своїм логіном/паролем.
- [ ] Надіслали тестову заявку з сайту → вона зʼявилась в адмінці **і** прийшла на пошту.
- [ ] Підключено постійний диск (`DATA_DIR`/`UPLOAD_DIR`) — дані не зникають після передеплою.
- [ ] (Бажано) Підключили власний домен у налаштуваннях хостингу.
- [ ] Змінили `ADMIN_PASSWORD` на надійний і не зберігаєте його в git.
